## 0. 글로벌 공통 규칙

모든 검수 에이전트 스킬에 공통 적용되는 규칙.

| 항목 | 규칙 |
|------|------|
| **언어** | 모든 답변은 한국어로만 작성, 한자는 단 한 글자도 사용 금지 |
| **출력 형식** | 검수 및 분석 결과는 반드시 표(Table)와 단계별 체크리스트로 출력 |
| **분석 원칙** | 추측 최소화, 입력된 텍스트/코드/이미지 설명 기반의 논리적 분석 결과 도출 |
| **역할 한계** | 직접 코드 수정 금지 — 분석, 역설계, 제안만 수행 |
| **단일 원본** | 산출물은 `_audit/` 폴더에만 저장 |

### 공통 신뢰도 분류 체계

모든 검수 에이전트 산출물에는 아래 신뢰도 분류를 적용한다.

| 상태 | 의미 | 사용 기준 |
|------|------|----------|
| `Confirmed` | 확인된 사실 | HTML/CSS/네트워크 데이터에서 직접 확인 가능 |
| `Likely` | 높은 가능성의 추정 | 복수의 간접 근거가 일치 |
| `Hypothesis` | 참고용 가설 | 단일 근거 또는 패턴 기반 추론 |
| `Unknown` | 증거 부족 | 클라이언트 측에서 확인 필요 |

산출물 보고서의 분석 항목에는 아래 공통 칼럼을 포함한다.

| 필드 | 의미 |
|------|------|
| 항목 | 분석 대상 |
| 관찰 내용 | 실제로 확인한 내용 |
| 근거 | HTML, CSS, 네트워크, 화면 설명 등 출처 |
| 신뢰도 | High / Medium / Low |
| 상태 | Confirmed / Likely / Hypothesis / Unknown |
| 후속 확인 필요 | 제작 전 사람이 추가 검증해야 하는지 여부 |

### 외부 사이트 분석 보안 및 운영 정책

외부 URL, 타사 사이트를 분석할 때 반드시 준수해야 하는 규칙.

| # | 규칙 |
|---|------|
| 1 | 공개 접근 가능한 페이지만 분석. 로그인, 결제, 관리자 화면은 사용자 명시 승인 없이 분석 금지 |
| 2 | 쿠키, 세션 토큰, API 키 등 인증 정보 저장 금지 |
| 3 | 네트워크 로그, 캡처 이미지에 개인정보(PII)가 포함된 경우 마스킹 후 저장 |
| 4 | 대상 사이트의 `robots.txt`, 이용약관, 요청 제한(rate limit) 준수 |
| 5 | 자동 수집 도구 사용 시 과도한 요청 금지 (최소 1초 간격 권장) |
| 6 | 수집 데이터는 분석 목적으로만 사용. 제3자 공유 금지 |

## 1. 위치 및 역할 구분

이 팀은 기존 제작 에이전시 시스템과 **독립적으로 운영**되지만, Step 5에서 제작 팀의 Agent 1 (PM)에게 인계하는 방식으로 **연동**된다.

```
[기존 제작 팀 — agent-ai-agent-plan.md]
  Agent 1: PM          (/pm)       ← _agency/client-brief.md 읽음
  Agent 2: 디자이너    (/design)
  Agent 3: FE 개발자   (/fe)
  Agent 4: BE 개발자   (/be)
       ↑
  [Step 6 인계 포인트]  ← _agency/client-brief.md 자동 생성
       |                  (선택: _audit/db-report.md → /contract 참고)
       |                  (선택: _audit/report.md → /qa-check 참고)
[검수 팀 — 본 문서]
  Agent 5: UI/UX 리서처          (/audit-ux)
  Agent 6: 정보 아키텍트          (/audit-ia)
  Agent 7: 기술 스택 진단가       (/audit-tech)
  Agent 8: 데이터베이스 유추가    (/audit-db)
  오케스트레이터: 전체 조율       (/audit)
```

## 3. 에이전트 상세 스킬셋

### 🕵️ Agent 5 — UI/UX 리서처 및 컴포넌트 수집가 (`/audit-ux`)

**역할:** 디자인 토큰(포인트 컬러, 폰트) 추출, 주요 UI 컴포넌트 패턴 분석

**주요 스킬:** 웹 디자인 역설계, 여백 및 레이아웃 규칙 파악, 접근성 검수

**System Prompt:**
```
너는 예리한 눈을 가진 시니어 웹 디자인 리서처야.
제공된 웹사이트의 화면 설명, 코드 조각, 또는 스크래핑 데이터를 바탕으로
브랜드 메인 컬러(HEX 코드), 서브 컬러, 배경 컬러, 그리고 사용된 타이포그래피 규칙을
찾아내어 표 형태로 정리해.
버튼, 입력창, 내비게이션 바 등 자주 반복되는 주요 UI 컴포넌트의 스타일 특징을 수집하고,
기존 디자인의 아쉬운 점과 개선 방향을 체크리스트로 제시해.
```

**입력:**
- `_audit/target.md` (분석 대상 정보)
- 대상 사이트 HTML/CSS 코드 조각
- 페이지 화면 캡처 설명 또는 이미지
- 스크래핑 데이터

**의존성:** 없음 (첫 단계)

**산출물 (`_audit/ux-report.md`) 구조:**

```
■ 디자인 토큰
  - 컬러 팔레트 표 (역할, HEX, 사용 위치)
  - 타이포그래피 표 (폰트명, 용도, 사이즈/굵기)
  - 여백 및 그리드 규칙

■ UI 컴포넌트 명세
  - 컴포넌트별 스타일 특징 표
  - 반복 패턴 및 일관성 여부

■ 접근성 체크리스트
  - 색상 대비 비율 충족 여부
  - alt 텍스트, 포커스 관리 등

■ 개선 제안 체크리스트
  - 아쉬운 점과 리뉴얼 방향
```

### 🗺️ Agent 6 — 정보 아키텍트 (`/audit-ia`)

**역할:** 사이트 카테고리(IA) 파악, 메뉴 구조 검수, 사용자 여정 점검

**주요 스킬:** 사이트맵 역추적, 콘텐츠 계층화, SEO 메타데이터 분석

**System Prompt:**
```
너는 웹사이트 정보 구조 분석 전문가야.
제공된 사이트의 메뉴 텍스트나 링크 목록을 꼼꼼하게 확인해서
현재 웹사이트의 전체 카테고리 트리와 사이트맵을 계층적인 체크리스트 형태로 다시 그려내.
사용자가 원하는 정보에 도달하기까지의 과정이 직관적인지 평가하고,
불필요하게 쪼개져 있거나 통합이 필요한 메뉴 카테고리 개선안을 표로 정리해서 알려줘.
```

**입력:**
- `_audit/ux-report.md` (UI 컴포넌트 패턴 — 네비게이션 구조 참조)
- 전체 메뉴 텍스트
- 하위 링크 목록
- 각 페이지 URL 구조

**의존성:** Step 1 (`/audit-ux`) 완료

**산출물 (`_audit/ia-report.md`) 구조:**

```
■ 현재 사이트맵 (계층 체크리스트)
  - 1depth: 메인 카테고리
  - 2depth: 서브 카테고리
  - 3depth 이하: 상세 페이지

■ 사용자 여정 평가
  - 주요 전환 경로 (예: 방문 → 서비스 확인 → 문의)
  - 탐색 단계 수 및 직관성 평가

■ IA 개선 제안 표
  - 현재 구조 / 문제점 / 개선안 / 우선순위

■ SEO 메타데이터 현황
  - 주요 페이지별 title, description 현황
```

### 🏗️ Agent 7 — 기술 스택 및 인프라 진단가 (`/audit-tech`)

**역할:** 프론트엔드 프레임워크, 로딩 성능, 호스팅 환경 등 기술적 요소 파악

**주요 스킬:** 웹 성능 최적화 진단, 소스코드 헤더 분석, 서드파티 스크립트 탐지

**System Prompt:**
```
너는 베테랑 웹 아키텍트이자 데브옵스 엔지니어야.
전달받은 사이트의 크롬 개발자 도구 결과(네트워크 탭 데이터, 엘리먼트 소스 등)를 바탕으로,
해당 사이트가 어떤 자바스크립트 프레임워크와 라이브러리를 사용해서 만들어졌는지 분석해.
로딩 속도를 저하시키는 무거운 스크립트나 이미지 요소를 찾아내고,
클라이언트 환경에서 파악할 수 있는 인프라 특징을 기술 스택 표로 깔끔하게 정리해.
```

**입력:**
- `_audit/ia-report.md` (분석 대상 페이지 목록 참조)
- 크롬 개발자 도구 네트워크 탭 데이터 (PII/토큰 제거 후)
- 페이지 HTML 소스 (`<head>` 영역)
- 사용된 스크립트 목록

**의존성:** Step 2 (`/audit-ia`) 완료

**산출물 (`_audit/tech-report.md`) 구조:**

> 보안 참고: 네트워크 탭 데이터에 포함된 인증 토큰, 세션 쿠키, API 키는 입력 전 반드시 제거

```
■ 기술 스택 표
  - 레이어 / 감지된 기술 / 근거 / 신뢰도

■ 성능 진단
  - 로딩 속도 저하 요인 목록
  - 무거운 리소스 (이미지, 스크립트) 상위 5개
  - Core Web Vitals 예상 점수 (LCP, CLS, INP)

■ 서드파티 스크립트 목록
  - 스크립트명 / 용도 / 성능 영향 / 제거 가능 여부

■ 인프라 특징 (클라이언트에서 파악 가능한 범위)
  - 서버 헤더 정보 (Server, X-Powered-By 등)
  - CDN 사용 여부
  - HTTPS 설정 상태
```

### 💾 Agent 8 — 데이터베이스 및 데이터 흐름 유추가 (`/audit-db`)

**역할:** 화면의 기능(게시판, 결제, 회원가입 등)을 바탕으로 기존 데이터베이스 모델링(ERD) 역설계

**주요 스킬:** 데이터 스키마 유추, API 페이로드 분석, 관계형/비관계형 데이터 흐름 파악

**System Prompt:**
```
너는 데이터 모델링 및 리버스 엔지니어링 전문가야.
기존 사이트에 구현된 기능(예: 상품 목록 필터링, 회원 마이페이지, 리뷰 게시판 등)과
화면 입력 항목들을 보고, 백그라운드에서 동작하고 있을 데이터베이스 테이블 구조(ERD)를
표 형태로 역설계해.
프론트엔드와 서버 간에 어떤 데이터(JSON)가 오가는지 예상되는 핵심 API 엔드포인트와
요청/응답 데이터 구조를 작성해.
```

**입력:**
- `_audit/ia-report.md` (사이트맵 + 기능 목록)
- `_audit/tech-report.md` (API 호출 증거 — 필수 참조)
- 화면의 주요 기능 및 입력 폼 항목 목록
- 네트워크 탭에서 확인된 API 호출 (선택)

**의존성:** Step 2 (`/audit-ia`) + Step 3 (`/audit-tech`) 완료

**입력 게이트 규칙:**
- `_audit/tech-report.md`에 API 호출 증거가 있으면 → 정상 분석
- API 호출 증거가 없거나 부족하면 → ERD/API 산출물을 `Hypothesis` 레벨로만 작성
- 게이트 판정 결과를 `_audit/db-report.md` 상단에 명시

**산출물 (`_audit/db-report.md`) 구조:**

```
■ 입력 게이트 판정
  - API 증거 유무: 있음 / 부족 / 없음
  - 산출물 신뢰 등급: 정상 분석 / Hypothesis 한정

■ 추정 ERD (테이블 구조 표)
  - 테이블명 / 주요 컬럼 / 데이터 타입 / 관계

■ 테이블 간 관계 다이어그램 (텍스트 형식)
  - User (1) ─── (N) Order
  - Order (1) ─── (N) OrderItem
  - 등

■ 핵심 API 엔드포인트 유추 표
  - Method / Endpoint / 요청 파라미터 / 응답 데이터 구조

■ 리뉴얼 시 주의사항
  - 데이터 마이그레이션 고려 항목
  - 기존 구조의 문제점 및 개선 방향
```

```
[타겟 사이트 URL 또는 데이터 입력]
        |
   /audit 오케스트레이터 실행
        |
   ┌────────────────┐
   │  _audit/       │
   │  status.md     │ ← 단계별 진행 상태 자동 기록
   └────────────────┘
        |
  ┌─────▼──────────────────────────────────────────────┐
  │ Step 0: /audit — 분석 대상 등록 (target.md 생성)    │
  │ Step 1: /audit-ux — 시각 정보 수집                  │
  │ Step 2: /audit-ia — 구조 해체 (← Step 1 참조)      │
  │ Step 3: /audit-tech — 기술 분석 (← Step 2 참조)    │
  │ Step 4: /audit-db — 데이터 유추 (← Step 2+3, 게이트)│
  │ Step 5: /audit — 종합 보고서 (known gaps 포함)      │
  │ Step 6: /audit — client-brief.md 생성 → /pm 인계   │
  └────────────────────────────────────────────────────┘
```

### Step 0 — 분석 대상 등록 (담당: `/audit` 오케스트레이터)

- [ ] `_audit/` 폴더를 생성한다.
- [ ] `_audit/target.md`를 아래 템플릿으로 생성하고 사용자에게 작성을 안내한다.
- [ ] `_audit/status.md`를 초기화한다.

**`_audit/target.md` 템플릿:**

```markdown
# 분석 대상 정보

## 기본 정보
- **사이트명:** (예: 김인테리어)
- **URL:** (예: https://example.com)
- **사이트 유형:** (예: 기업 소개, 쇼핑몰, 포트폴리오)
- **분석 목적:** (예: 리뉴얼 제안서 작성, 경쟁사 벤치마킹)
- **고객 제공 권한:** (예: 공개 페이지만 / 관리자 접근 가능)

## 입력 데이터 (해당 항목 체크)
- [ ] 주요 페이지 HTML/CSS 코드 조각
- [ ] 페이지 화면 캡처 (이미지 또는 설명)
- [ ] 전체 메뉴 텍스트 및 링크 목록
- [ ] 크롬 개발자 도구 네트워크 탭 데이터 (민감 정보 제거)
- [ ] 페이지 HTML 소스 (<head> 영역)
- [ ] 스크래핑 데이터 폴더 경로: 

## 특이사항
(분석 시 참고할 사항이 있으면 기록)
```

### Step 1 — 시각 정보 수집 (담당: `/audit-ux`)

- [ ] 타겟 사이트의 핵심 페이지 HTML 또는 캡처 정보를 준비한다.
- [ ] `/audit-ux` 에이전트에게 전달한다.
- [ ] 포인트 컬러, 폰트, 공통 디자인 컴포넌트 명세서를 `_audit/ux-report.md`에 저장한다.

### Step 2 — 구조 해체 (담당: `/audit-ia`)

- [ ] 전체 메뉴 텍스트와 하위 링크 목록을 정리한다.
- [ ] `/audit-ia` 에이전트에게 전달한다.
- [ ] 현재 사이트맵 재구성 및 카테고리 통폐합 의견을 `_audit/ia-report.md`에 저장한다.

### Step 3 — 기술 분석 (담당: `/audit-tech`)

- [ ] 개발자 도구에서 헤더 정보, 사용된 스크립트 목록을 수집한다.
- [ ] `/audit-tech` 에이전트에게 입력한다.
- [ ] 프레임워크, 성능 진단, 서드파티 목록을 `_audit/tech-report.md`에 저장한다.

### Step 4 — 데이터 유추 (담당: `/audit-db`)

- [ ] **입력 게이트 점검:** `_audit/tech-report.md`에 API 호출 증거가 있는지 확인한다.
  - 있으면 → 정상 분석 모드
  - 없거나 부족하면 → `Hypothesis` 한정 모드로 전환
- [ ] Step 2의 사이트맵 + Step 3의 API 호출 증거를 함께 전달한다.
- [ ] `/audit-db` 에이전트가 추정 ERD, 핵심 API 엔드포인트를 `_audit/db-report.md`에 저장한다.
- [ ] 게이트 판정 결과를 보고서 상단에 명시한다.

### Step 5 — 종합 보고서 생성 (담당: `/audit` 오케스트레이터)

- [ ] Step 1~4의 모든 분석 결과를 통합한다.
- [ ] Blocked 또는 `Hypothesis` 한정 항목이 있으면 `known gaps` 섹션을 강제 포함한다.
- [ ] `_audit/report.md` (종합 보고서) 생성한다.

### Step 6 — 제작 팀 인계 (담당: `/audit` 오케스트레이터)

- [ ] `_audit/report.md`를 기반으로 `_agency/client-brief.md` 초안을 자동 생성한다.
  - 변환 규칙: 분석 결과에서 사업명, 핵심 기능, 페이지 목록, 기술 요구사항을 추출
  - `Hypothesis` 항목은 "추가 확인 필요"로 표기
- [ ] **제작 팀의 `Agent 1 (PM / /pm)`에게 인계한다.**
  - 인계 파일: `_agency/client-brief.md` (PM이 바로 읽을 수 있는 형식)
  - 참고 파일: `_audit/report.md` (상세 분석 원본)
- [ ] (선택) 리뉴얼 프로젝트인 경우, `/contract` 단계에서 `_audit/db-report.md`의 ERD 초안을 참고 입력으로 활용할 수 있음을 안내한다.
- [ ] (선택) `/qa-check` 단계에서 `_audit/report.md`의 접근성/성능 문제를 검증 항목에 포함할 것을 권장한다.

## 5. 산출물 폴더 구조

`/audit` 실행 시 생성되는 구조:

```
{project-name}/
├── _audit/
│   ├── status.md           ← 분석 진행 상태 (자동 관리)
│   ├── target.md           ← 분석 대상 정보 (URL, 입력 데이터, 권한 범위)
│   ├── ux-report.md        ← Agent 5 산출물 (컬러, 폰트, 컴포넌트)
│   ├── ia-report.md        ← Agent 6 산출물 (사이트맵, IA 개선안)
│   ├── tech-report.md      ← Agent 7 산출물 (기술 스택, 성능)
│   ├── db-report.md        ← Agent 8 산출물 (ERD, API 유추)
│   └── report.md           ← 종합 보고서 (known gaps 포함)
└── _agency/
    └── client-brief.md     ← Step 6에서 자동 생성 (제작팀 /pm 인계용)
```

### `_audit/status.md` 템플릿

```markdown
# Audit Pipeline Status

| 단계 | 스킬 | 상태 | 완료일 | 비고 |
|------|------|------|--------|------|
| 0. 대상 등록 | /audit | ⏳ 대기 | - | - |
| 1. UX | /audit-ux | ⏳ 대기 | - | - |
| 2. IA | /audit-ia | ⏳ 대기 | - | - |
| 3. Tech | /audit-tech | ⏳ 대기 | - | - |
| 4. DB | /audit-db | ⏳ 대기 | - | API 증거 게이트 |
| 5. 종합 보고서 | /audit | ⏳ 대기 | - | - |
| 6. 제작팀 인계 | /audit | ⏳ 대기 | - | client-brief.md 생성 |
```

상태값:
- `⏳ 대기` — 아직 시작하지 않음
- `🔄 진행중` — 현재 실행 중
- `✅ 완료` — 정상 완료
- `⛔ 차단` — 선행 단계 미완료 또는 입력 부족

## 6. 기존 `/qa-check`와의 역할 구분

| 구분 | `/qa-check` (기존 제작 팀) | `/audit-*` 팀 (본 문서) |
|------|--------------------------|------------------------|
| **목적** | 내가 만든 사이트 배포 전 검사 | 기존 운영 중 사이트 역설계 및 진단 |
| **대상** | 신규 개발 중인 코드 | 외부 URL / 타사 제작 완료 사이트 |
| **시점** | 배포 직전 (QA 게이트) | 영업 전 / 리뉴얼 계약 전 |
| **출력** | 배포 가능 여부 판단 | 제작팀 `/pm` 인계용 분석 보고서 |
| **코드 접근** | 항상 가능 (내 코드) | HTML/JS 공개 소스 범위 내 |
| **독립성** | 제작 파이프라인 내 단계 | 독립 실행 후 제작팀으로 인계 |
