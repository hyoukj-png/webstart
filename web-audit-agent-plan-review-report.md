# 기존 웹사이트 검수 및 분석 에이전트 팀 계획서 검토 보고서

> 대상 문서: [web-audit-agent-plan.md](/Volumes/sub_ssd/Works/Antigravity/WebStart/web-audit-agent-plan.md)  
> 작성일: 2026-04-02  
> 목적: 발견된 문제, 영향, 보완 방향, 개정 우선순위를 한 번에 정리한 검토 보고서

## 1. 요약

원본 계획서는 "기존 사이트를 역설계해서 제작 팀으로 인계한다"는 목적이 분명하고, 역할 분리도 직관적이다. 특히 UX, IA, Tech, DB를 별도 에이전트로 나눈 구조는 확장성도 좋다.

다만 현재 버전은 "분석용 개념 문서"로는 괜찮지만, 실제 운영 문서로 쓰기에는 몇 가지 중요한 빈틈이 있다.

- `/audit`에서 `/pm`으로 넘기는 인계 계약이 실제 기존 스킬 구조와 맞지 않는다.
- 추정이 많이 들어가는 단계에 신뢰도, 근거, 불확실성 표기가 강제되지 않는다.
- 외부 사이트 분석에 필요한 보안, 법적, 운영 가드레일이 빠져 있다.
- 단계별 재시도, 실패, 부분 실행을 관리할 상태 모델이 약하다.
- 기존 제작 파이프라인과 연결되는 실제 파일 흐름이 모호하다.

결론적으로, 현재 문서는 방향은 좋지만 "실행 가능한 운영 설계"로 보강이 필요하다.

## 2. 핵심 문제점 및 개선안

| 우선순위 | 문제점 | 영향 | 개선안 |
|---|---|---|---|
| High | `/audit` 결과를 `/pm`에 인계하는 계약이 실제 `/pm` 입력 규약과 맞지 않음 | Step 5에서 파이프라인이 끊기고, 제작 팀이 바로 착수할 수 없음 | `/audit`가 `_audit/report.md`와 별도로 `_agency/client-brief.md` 초안을 생성하거나, `/pm`이 `_audit/report.md`를 읽을 수 있도록 입력 계약을 확장 |
| High | DB/API 역설계 결과에 근거와 신뢰도 표기가 강제되지 않음 | 환각성 ERD, 잘못된 API 추정이 후속 설계와 개발 단계로 전파될 수 있음 | 모든 보고서에 `근거`, `신뢰도`, `추가 확인 필요`, `확정 여부` 필드를 공통 도입 |
| High | 외부 URL 분석에 대한 보안 및 법적 가드레일이 없음 | 로그인 우회, 개인정보 노출, 과도한 스크래핑 같은 운영 리스크 발생 가능 | 공개 페이지 한정, PII 마스킹, 쿠키/토큰 저장 금지, robots/약관 준수, rate limit 준수 규칙 추가 |
| Medium | `/audit-db`가 `/audit-tech`의 네트워크 증거 없이도 실행 가능하게 설계됨 | 가장 불확실한 단계가 가장 약한 입력으로 돌아가 결과 품질이 흔들림 | `/audit-db` 입력을 `IA + 기능 목록 + API 호출 증거`로 강화하고, 증거가 없을 때는 "가설 수준"으로만 출력 제한 |
| Medium | `status.md`가 자동 관리된다고만 되어 있고 상태 모델이 없음 | 부분 실행, 재시도, 실패 복구, 재개 흐름 구현이 어려움 | `Pending/In Progress/Done/Blocked`, 입력 출처, 시작/완료 시각, 실패 사유를 포함한 템플릿 정의 |
| Medium | 감사 산출물 루트와 제작 파이프라인 루트의 연결 방식이 모호함 | 독립 분석 이후 실제 어떤 파일을 어디로 복사 또는 변환할지 혼선 발생 | `_audit`와 `_agency` 간 연결 규칙을 문서에 명시하고, 오케스트레이터가 변환 산출물까지 생성하도록 규정 |
| Low | 문서 내 명칭과 기존 저장소 상태 사이에 자잘한 불일치가 있음 | 운영 중 신뢰도 저하, 사용자 혼란 증가 | 문서명 오타, 슬래시 커맨드 예시, `install.sh` 설명을 실제 저장소 상태에 맞게 정리 |

## 3. 상세 검토

### 3.1 인계 계약 문제

현재 문서는 Step 5에서 `_audit/report.md`를 `/pm`에 넘겨 리뉴얼 기획안 작성을 시작한다고 적고 있다. 하지만 기존 `/pm` 스킬은 `_agency/client-brief.md`가 먼저 존재해야만 작동한다.

즉 현재 구조에서는 아래 문제가 생긴다.

1. 검수 팀은 `_audit/report.md`를 만든다.
2. 제작 팀 `/pm`은 `_agency/client-brief.md`를 찾는다.
3. 파일이 없으면 작업을 멈춘다.

이건 가장 먼저 해결해야 할 설계 결함이다.

### 3.2 증거 기반 분석 규칙 부족

문서 상단에는 추측 최소화 원칙이 있지만, 실제 Agent 7과 Agent 8의 산출물 정의에는 불확실성을 통제하는 장치가 없다. 특히 DB, API, 기술 스택 추정은 화면과 일부 네트워크 데이터만으로는 확정할 수 없는 경우가 많다.

따라서 "분석 결과"와 "가설"을 분리하지 않으면, 이후 제작 팀이 이를 사실처럼 받아들이게 된다.

권장 공통 규칙:

- 확인된 사실은 `Confirmed`
- 높은 가능성의 추정은 `Likely`
- 참고용 가설은 `Hypothesis`
- 증거 부족 항목은 `Unknown`

### 3.3 외부 사이트 분석 운영 리스크

본 문서는 외부 URL, 타사 제작 완료 사이트, 스크래핑 데이터, 개발자 도구 결과를 다룬다. 그런데 실제 운영에 필요한 최소한의 제한이 없다.

반드시 들어가야 할 규칙:

- 로그인, 결제, 관리자 화면은 사용자 제공 권한 없이는 분석 금지
- 쿠키, 토큰, 개인 식별 정보는 저장 금지
- 첨부 이미지나 네트워크 로그에 개인정보가 있으면 마스킹 후 저장
- 사이트 약관, robots, 요청 제한을 위반하는 자동 수집 금지

### 3.4 상태 관리와 부분 실행 설계 부족

`/audit --step=ux` 또는 `/audit --full` 같은 사용 예시는 좋다. 하지만 이를 뒷받침하는 상태 머신 정의가 없다.

지금 상태로는 아래 질문에 답할 수 없다.

- Step 2가 실패하면 Step 1 결과는 유지되는가
- Step 3만 재실행할 때 기존 Step 3 결과는 덮어쓰는가
- Step 4가 Blocked면 Step 5는 요약 보고서를 생성하는가
- 입력 데이터가 바뀌면 어느 단계부터 다시 돌아가야 하는가

운영 문서라면 이 정도는 명시되어야 한다.

## 4. 보완 방향 계획

### 4.1 구조 보완

1. **인계 산출물 이중화**
   - `_audit/report.md`: 사람이 읽는 종합 분석 보고서
   - `_agency/client-brief.md`: `/pm`이 바로 읽을 수 있는 제작 착수용 브리프 초안

2. **오케스트레이터 책임 확대**
   - 단순 취합이 아니라 "분석 결과를 제작팀 입력 형식으로 변환"하는 책임을 `/audit`에 부여
   - Step 5 완료 조건에 `_agency/client-brief.md` 생성 포함

3. **단계별 게이트 추가**
   - `/audit-db` 실행 전 `/audit-tech`의 API 근거 유무 점검
   - 근거 부족 시 ERD/API는 `Hypothesis` 레벨로만 작성
   - Step 5에서 Blocked 항목이 있으면 `known gaps` 섹션을 강제 포함

### 4.2 공통 출력 규격 보완

모든 보고서에 아래 공통 표준 칼럼을 넣는 것이 좋다.

| 필드 | 의미 |
|---|---|
| 항목 | 분석 대상 |
| 관찰 내용 | 실제로 확인한 내용 |
| 근거 | HTML, CSS, 네트워크, 화면 설명 등 |
| 신뢰도 | High / Medium / Low |
| 상태 | Confirmed / Likely / Hypothesis / Unknown |
| 후속 확인 필요 | 제작 전 사람이 추가 검증해야 하는지 |

특히 `_audit/db-report.md`, `_audit/tech-report.md`에는 필수다.

### 4.3 상태 관리 보완

`_audit/status.md`에 아래 항목을 포함하는 것을 권장한다.

```markdown
# Audit Pipeline Status

| 단계 | 스킬 | 상태 | 입력 출처 | 시작일 | 완료일 | 비고 |
|------|------|------|----------|--------|--------|------|
| 1. UX | /audit-ux | ✅ 완료 | URL + screenshot | 2026-04-02 | 2026-04-02 | - |
| 2. IA | /audit-ia | ⏳ 대기 | - | - | - | - |
| 3. Tech | /audit-tech | ⏳ 대기 | - | - | - | - |
| 4. DB | /audit-db | ⏳ 대기 | - | - | - | API 근거 필요 |
| 5. Report | /audit | ⏳ 대기 | Step 1~4 결과 | - | - | - |
```

상태값은 최소 아래 4가지를 권장한다.

- `⏳ 대기`
- `🔄 진행중`
- `✅ 완료`
- `⛔ 차단`

### 4.4 보안 및 운영 정책 보완

새 섹션으로 아래 항목을 넣는 것을 권장한다.

1. 공개 접근 가능한 데이터만 분석
2. 인증이 필요한 영역은 사용자 명시 승인 없이는 제외
3. 네트워크 로그에 포함된 민감 데이터 저장 금지
4. 개인정보가 포함된 캡처와 데이터는 마스킹 후 저장
5. 자동 수집은 대상 사이트 정책과 요청 제한을 준수

### 4.5 기존 제작 파이프라인 연동 보완

현재 가장 현실적인 방식은 아래 둘 중 하나다.

1. `/audit`가 `_agency/client-brief.md`를 자동 생성
   - 장점: 기존 `/pm` 스킬 수정이 거의 필요 없음
   - 단점: `/audit` 책임이 조금 커짐

2. `/pm` 스킬을 확장해 `_audit/report.md`도 입력으로 허용
   - 장점: 인계가 자연스러움
   - 단점: 기존 제작 파이프라인 계약을 수정해야 함

우선순위는 1번이 더 안전하다.

## 7. 최종 판단

현재 계획서는 아이디어 자체는 좋고, 제작 에이전시 시스템의 사전 영업/진단 레이어로 확장하려는 방향도 타당하다.

하지만 그대로 구현에 들어가면 가장 먼저 Step 5 인계에서 막히고, 그 다음에는 추정 결과의 신뢰도 문제와 외부 사이트 분석 정책 부재가 운영 리스크로 드러날 가능성이 높다.

따라서 다음 개정판의 목표는 기능 추가보다 "연결 계약 명확화"와 "불확실성 통제"에 두는 것이 좋다. 이 두 가지만 먼저 닫아도 문서 완성도가 크게 올라간다.
